Resumo:

O objetivo deste documento é prover os recursos necessários para que o procedimento de revisão de código seja implementado e regrado. Aqui constam convenções que serão adotadas e procedimentos que deverão seguidos por todos os desenvolvedores.

Introdução

Todo código deverá ser avaliado por um revisor de código. Neste caso o papel do revisor é avaliar se o desenvolvedor produziu código que poderá atrapalhar a semântica do projeto ou a manutenção num futuro próximo, também deve verificar se o código produzido é eficiente e atual com relação aos procedimentos utilizados no Laboratório. Boa parte das técnicas empregadas na revisão de código derivam das técnicas desenvolvidas no mozilla.org, veja:

http://www.mozilla.org/hacking/reviewers.html
http://www.mozilla.org/projects/seamonkey/rules/code_review.html

Também cabe ao revisor ensinar, se for o caso, o desenvolvedor novas técnicas e/ou mostrar novas soluções. Uma discussão sobre eventuais mudanças no código sempre deverá acontecer entre o revisor e o desenvolvedor para que este tome conhecimento de o quê e o porquê seu código sofreu mudanças, difundindo assim o conhecimento entre os desenvolvedores.

Procedimentos de revisão de código

Os procedimentos serão os mesmos adotados pelo mozilla.org. Uma lista de e-mail será criada, revisores@bit.fmrp.usp.br, onde cópias dos e-mails relacionados aos procedimentos de revisão serão postados. Ao receber a aprovação de revisão o desenvolvedor deverá realizar um commit adicionando o e-mail do revisor no log do CVS. Exemplo: r:mavcunha@bit.fmrp.usp.br, pode-se omitir o host para membros do Laboratório, como r:mavcunha.

Para iniciar o procedimento de revisão o desenvolvedor após alterar seu código de trabalho e estar interessado e realizar o commit de seu código, deverá solicitar, através de e-mail com cópia para revisores@bit.fmrp.usp.br, a um revisor que este dê sua aprovação sobre o commit. Para enviar a informação, basta o desenvolvedor utilizar o diff de seu código em relação a árvore atual.

Comando: cvs diff file file.patch

Este comando criará um arquivo que será utilizado pelo revisor para atualizar sua cópia do código e verificar as mudanças realizadas pelo desenvolvedor. O revisor por sua vez executará um patch no arquivo de trabalho.

Comando: patch original_file patch_file

O revisor, ao terminar de olhar o código, deverá responder o e-mail comunicando a aprovação ou não e suas justificativas.

Data e carga de revisores

O desenvolvedor ao escolher um revisor em particular provavelmente tem razões para fazê-lo. No entanto o revisor pode estar muito ocupado e não poderá fazer a revisão do código. Neste caso o revisor deverá replicar o e-mail para a lista revisores@bit.fmrp.usp.br comunicando sua falta de tempo, neste caso qualquer outro revisor pode assumir a tarefa e revisar o código.

O desenvolvedor também pode restringir a revisão do código com uma data máxima, que deve ser razoável para que algum revisor possa realizar a tarefa.

Exceções a revisão

Um desenvolvedor pode requisitar um commit imediato. Isto é, colocar o código diretamente no CVS sem que haja a revisão do mesmo. Isto será possivel mas a revisão será adiada e não cancelada. As situações onde um desenvolvedor pode requisitar commit imediato são:

  • BUG que inviabiliza algum serviço já aberto aos usuários.
  • Trabalho de fim de semana que esteja dependendo de revisão para proseguir.
  • Urgência no projeto sem a presença de revisores suficientes.

Nestes casos o desenvolvedor pode aplicar o código ao projeto em questão sem uma revisão e deve marcar no log do CVS r:urgente. Assim uma revisão posterior será realizada assim que um revisor puder. Os revisores deverão priorizar qualquer commit realizado e marcado como urgente, evitando o acúmulo de código não revisado no CVS.
